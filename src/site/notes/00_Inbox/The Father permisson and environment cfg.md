# 环境变量与配置继承指南

## 1. 核心概念：什么是环境变量？

**环境变量（Environment Variables）是“程序启动时读取的外部参数/约定入口”，而非 API。**

- **API**：程序运行时主动调用的外部接口（动态交互）。
- **环境变量**：程序启动瞬间，先观察外部环境提供的“启动说明书”（静态注入）。

### 常见用途：
- `PATH`：定义系统搜索**可执行程序**的路径顺序。
- `http_proxy` / `https_proxy`：强制程序遵循的**网络代理**路径。
- `OPENAI_API_KEY`：程序认证所需的**密钥**。
- `NODE_ENV`：定义当前是开发环境（development）还是生产环境（production）。

---

## 2. 继承逻辑（父系权限）

软件启动时会**自动继承**其父进程（启动它的环境）的环境变量，这就是所谓的“父系权限”。

- **逻辑示例**：如果你在 `C:/Admin/` 路径下的终端启动了 VS Code，VS Code 将继承该终端环境中已加载的所有变量设置。
- **排查思路**：当遇到变量错误或需要绕过官方配置时，应检查：
    1. **系统级/用户级环境**（全局）。
    2. **终端会话环境**（当前窗口临时变量）。
    3. **父应用注入环境**（如从 IDE 启动调试器）。

---

## 3. 配置层级与优先级链

开发者工具通常遵循一套严格的优先级，从“最具体”到“最通用”：

### 常见的优先级模型：
1. **命令行参数 (Runtime Flags)**：启动指令中带的参数（最高优先级）。
2. **本地/项目配置 (.settings / .env)**：存在于项目目录下的配置文件（如 `.env`, `settings.json`）。
3. **环境变量 (Env Vars)**：系统或 Shell 中定义的变量。
4. **全局用户配置 (Global Config)**：位于用户家目录（如 `~/.config`）的通用设置。
5. **程序默认值 (Defaults)**：代码硬编码的底线设置（最低优先级）。

> [!TIP]
> **注意**：部分软件会采用 `配置文件 > 环境变量` 的逻辑，具体取决于软件的设计哲学。

---

## 4. 实践建议

### 哪些软件需要配置？
- **开发工具链**：Node/Python/Go 编译器与运行环境。
- **基础设施**：Docker/容器应用、代理/网络工具。
- **云端与 AI**：云服务 SDK、AI API/CLI 工具（如 OpenAI/Anthropic CLI）。

### 开发者必经之路
在使用任何 AI 工具或开发者套件前，**养成先查看其 `config` 或 `.env.example` 文件的习惯**。深入了解其变量优先级，往往能发现隐藏的高级功能或解决复杂的环境冲突。
